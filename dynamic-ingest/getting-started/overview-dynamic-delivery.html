La diffusion de contenu multimédia sur différents appareils (sur ordinateur de bureau, mobile et téléviseurs connectés) nécessite un mélange de technologies en constante évolution :

  • Codecs audio et vidéo
  • Différents types d'emballages pour le streaming vidéo
  • Divers schémas de protection du contenu pour protéger le contenu contre la copie

Sans l'utilisation de Dynamic Delivery, le streaming multiplateforme nécessite souvent de stocker le même contenu plusieurs fois pour prendre en charge la livraison sur toutes vos plateformes cibles. La livraison dynamique permet aux clients de Brightcove de fournir des médias à une gamme d'appareils sans les coûts et les frais généraux liés au reconditionnement et au stockage du contenu plusieurs fois pour chaque appareil et cas d'utilisation pris en charge. La livraison dynamique crée le package de contenu optimal juste à temps pour la livraison et la diffusion en continu sur tous les appareils et cas d'utilisation pris en charge. Cela aide également à pérenniser votre architecture de livraison à mesure que la technologie et votre public évoluent.

Avec Dynamic Delivery, les clients fournissent un seul fichier vidéo (mezzanine) de haute qualité. Au cours du processus d'acquisition, nous créons plusieurs niveaux de qualité de ce fichier adaptés à une gamme de tailles d'écran et de conditions de réseau différentes. Lorsqu'un spectateur appuie sur play, Video Cloud utilise des informations sur l'appareil demandeur pour déterminer le type de technologie de streaming attendu - par exemple, le type d'empaquetage de flux (HLS vs. DASH) (ou MP4 si l'appareil ne prend en charge aucun type de HTTP streaming), le DRM requis (par exemple, Widevine, Fairplay, Playready) et le contenu auxiliaire tel que les sous-titres codés ou les pistes audio alternatives. La livraison dynamique utilise ensuite ces informations pour livrer le package de contenu approprié (c'est-à-dire le type de flux) pour l'appareil demandeur en temps réel.

Présentation de l'architecture

  1. Lorsqu'un fichier source est téléchargé, le processus de transcodage utilise le profil d'ingestion sélectionné pour déterminer l'ensemble de rendus à créer.
  2. Les rendus sont stockés sous la forme d'un ensemble de fichiers MP4 contigus et fragmentés de divers débits binaires et résolutions qui sont indépendants de tout format de livraison spécifique.
  3. Lorsqu'une demande est faite pour une vidéo, le processus de livraison dynamique effectue un emballage juste à temps ou JIT, ce qui signifie qu'il emballera dynamiquement le contenu dans le format approprié en fonction des exigences spécifiées. Cet emballage prendra en considération le format de livraison spécifique requis, les pistes audio et de sous-titres nécessaires, ainsi que tout cryptage/DRM requis.
  4. Lorsque l'appareil de lecture utilise le manifeste renvoyé par l'API de lecture pour demander la lecture des médias, nous effectuons un emballage juste à temps sur les fichiers MP4 fragmentés pour fournir le format de segment, la durée et le cryptage appropriés. Ces fragments sont mis en cache sur un CDN afin que la diffusion vidéo soit toujours très performante.

Le processus de livraison dynamique a été conçu pour être un service entièrement géré et basé sur le cloud. Tous les composants s'exécutent sur une infrastructure cloud dans plusieurs régions pour assurer la haute disponibilité, la redondance et l'évolutivité.

Quels formats vidéo produit Dynamic Delivery ?

La livraison dynamique produit les sources suivantes selon les besoins :

  • HLS (versions 3, 4 et 5)
  • Lisse
  • TIRET
  • MP4 (si des rendus progressifs sont spécifiés dans le profil d'ingestion)

Les packages DRM suivants sont créés (en fonction de la protection du contenu pour laquelle votre compte est activé) :

  • HLS + Fairplay
  • DASH + Widevine
  • DASH + Playready
  • Lisse + Prêt à jouer

Processus de migration

Depuis le mardi 17 octobre 2017, tous les nouveaux comptes Video Cloud créés sont configurés pour utiliser Dynamic Delivery par défaut. Cela inclut les comptes d'essai, les comptes pour les nouveaux éditeurs et les nouveaux sous-comptes pour les éditeurs existants. Avec ce changement, tous les nouveaux éditeurs et clients d'essai bénéficieront automatiquement de toutes les fonctionnalités de Dynamic Delivery.

Pour les clients existants, la migration est un processus en deux étapes. Premièrement, les nouveaux téléchargements sont transférés pour utiliser Dynamic Delivery (migration en aval d'un jour) et deuxièmement, le contenu existant est migré de la plate-forme de livraison héritée vers Dynamic Delivery (migration de catalogue arrière.)

Migration d'un jour

La première étape pour passer à la livraison dynamique consiste à transférer les nouveaux téléchargements pour utiliser la livraison dynamique. L'activation de votre compte pour la livraison dynamique est une tâche simple qui peut être effectuée par le service client. Il n'y a aucun danger à activer un compte Video Cloud existant pour la diffusion dynamique car il ne modifie aucun contenu existant. Seul le contenu explicitement ingéré dans Dynamic Delivery utilisera le nouveau système.

Du point de vue du flux de travail, quelques éléments apparaîtront différemment dans Video Cloud Studio. Tout d'abord, lors du téléchargement de vidéos à l'aide du module de téléchargement, vous verrez désormais les profils d'intégration de la diffusion dynamique dans votre compte.

Le module Média affichera également une icône de nuage vert () à côté des titres vidéo ingérés à l'aide d'un profil de diffusion dynamique. Les vidéos ingérées à l'aide d'un ancien profil d'ingestion affichent une icône grise ().

De plus, le panneau Sources vidéo du module multimédia apparaîtra légèrement différent en raison du stockage des rendus dans un état indépendant du format.

Fonctionnellement, tout le reste reste le même. Vous pouvez télécharger via tous les mécanismes standard (Video Cloud Studio, API d'ingestion dynamique, connecteurs CMS, etc.) et la lecture reste inchangée sur tous vos appareils de lecture standard.

Une fois qu'une vidéo a été ingérée à l'aide de Dynamic Delivery, si la vidéo est retranscodée, un profil Dynamic Delivery doit être sélectionné. Si vous souhaitez utiliser un ancien profil d'ingestion, la vidéo doit être rechargée.

Noter: La livraison dynamique introduit le concept d'ingestion et de livraison régionalisées. Dans les systèmes de livraison précédents, tout le traitement de l'ingestion et l'origine de la livraison résidaient aux États-Unis. Avec Dynamic Delivery, Brightcove peut configurer un compte où l'ingestion a lieu et où réside l'origine éditeur par éditeur pour de meilleures performances. Tous les comptes Video Cloud seront par défaut sur US-East mais peuvent être reconfigurés dans votre région locale. Notez qu'il y a des implications en termes de coûts lorsque vous déménagez dans une autre région. Veuillez contacter votre gestionnaire de compte Brightcove pour plus d'informations.

Migration de catalogue arrière

Une fois que vous avez de nouvelles acquisitions à l'aide de Dynamic Delivery, nous pouvons alors migrer votre catalogue de retour. Cela peut être fait de l'une des deux manières suivantes :

Outil de migration Transmux

L'outil de migration transmux déplace le contenu Video Cloud de la diffusion héritée vers la diffusion dynamique sans retranscodage. Cela vous permet de migrer votre contenu beaucoup plus rapidement et plus facilement que le retranscodage du contenu. En tant que tel, il devrait être la méthode de migration préférée pour la plupart des clients. L'outil transmux transformera les rendus existants dans votre compte pour qu'ils soient compatibles avec Dynamic Delivery. Pour migrer votre contenu via l'outil transmux, veuillez contacter votre gestionnaire de compte.

Re-transcoder

Il y a quelques critères qui rendent un titre inéligible pour migrer via l'outil transmux et en tant que tel, il faudra que les titres soient ré-encodés pour les migrer. Les titres devront être réencodés si :

  • Vous souhaitez utiliser l'encodage contextuel
  • Vous souhaitez faire de l'insertion d'annonces côté serveur (SSAI) mid-roll
  • Votre titre n'a pas de rendus HLS
  • Votre titre a actuellement des DRM (HLSe est acceptable)

Si vous devez retranscoder des titres, vous devrez soit gérer ce processus vous-même (en suivant nos meilleures pratiques), soit vous engager avec votre équipe de compte pour des services de migration payants. Noter: Il est possible de transmuxer certains titres au sein d'un compte et de retranscoder d'autres.

Questions fréquemment posées

Qu'est-ce que la livraison dynamique ?

Dynamic Delivery est un ensemble de technologies qui constituent la plate-forme de diffusion multimédia de nouvelle génération de Brightcove. La livraison dynamique est conçue pour répondre aux besoins actuels et futurs des éditeurs et deviendra la base de la façon dont tous les produits Brightcove livrent les médias.

Quels sont les avantages de la livraison dynamique ?

  • Une plus grande flexibilité et agilité pour la livraison - Le contenu est emballé de manière dynamique pour être livré à un certain nombre de points de terminaison (navigateurs Web, smartphones, tablettes, téléviseurs connectés et boîtiers de streaming). Dynamic Delivery est conçu pour être capable d'utiliser plusieurs CDN pour fournir du contenu, augmentant ainsi le choix des clients.
  • Épreuvage futur - Étant donné que le contenu est conditionné de manière dynamique, Brightcove peut ajouter de nouveaux formats à la volée. Lorsqu'une nouvelle version de HLS, DASH, etc., ou même un nouveau format sort, nous ajoutons simplement une prise en charge au niveau de la plate-forme et les éditeurs peuvent commencer à l'utiliser sans retranscoder le contenu existant.
  • Empreinte de stockage réduite - Les actifs de contenu sont stockés une fois et les rendus sont créés en aval pour être livrés en cas de besoin au lieu de stocker le contenu plusieurs fois.
  • Performances améliorées grâce à la distribution géographique - Dynamic Delivery est conçu pour une architecture hautement disponible et distribuée au niveau régional. Les éditeurs bénéficient de performances de lecture améliorées et d'une résilience accrue.
  • Prise en charge intégrée de l'insertion d'annonces côté serveur (SSAI) - La diffusion dynamique permettra d'insérer des annonces ou d'autres contenus à partir du serveur. Cela offre une expérience de lecture transparente pour les téléspectateurs et donne aux éditeurs une résilience accrue contre les bloqueurs de publicités, le tout depuis Video Cloud.

Y a-t-il des considérations de prix avec Dynamic Delivery ?

Non, l'accès à Dynamic Delivery est inclus dans chaque licence de plate-forme de base. Certaines fonctionnalités de la livraison dynamique, notamment l'insertion d'annonces côté serveur (SSAI) et l'encodage contextuel (CAE), sont des modules complémentaires avec des coûts associés. Vérifiez auprès de votre gestionnaire de compte pour plus de détails sur les prix.

Dynamic Delivery prend-il en charge les éditeurs BYO CDN ?

La livraison dynamique permet aux éditeurs de tirer parti de leurs propres relations CDN ; cependant, toutes les ressources vidéo doivent être gérées par Brightcove Origin. Veuillez contacter votre gestionnaire de compte pour plus d'informations.

Dans quelles régions Dynamic Delivery prend-il en charge l'ingestion et l'origine ?

Dynamic Delivery prend en charge l'ingestion et l'origine dans les régions AWS suivantes :

  • us-east-1 (Virginie) - par défaut pour tous les comptes
  • eu-ouest-1 (Dublin)
  • ap-nord-est-1 (Tokyo)
  • ap-sud-est-1 (Singapour)
  • ap-sud-est-2 (Sydney)

Notez que le déplacement de l'ingestion et de l'origine vers une autre région a des implications financières. Veuillez contacter votre gestionnaire de compte Brightcove pour plus d'informations.

Tous les comptes Brightcove sont-ils sur Dynamic Delivery ?

Oui, tous les nouveaux éditeurs (Video Cloud et Once) seront configurés avec Dynamic Delivery. Tous les comptes existants ont été migrés vers Dynamic Delivery et le système d'acquisition hérité a été désactivé.

Certains de mes rendus HLS ont un débit binaire plus élevé (selon l'API CMS) que ce qui est spécifié dans le profil d'ingestion - pourquoi ?

Ce qui est spécifié dans le profil d'ingestion est le vidéo débit binaire. Ce qui est rapporté par l'API CMS est le débit vidéo + le débit audio + une surcharge de 10 % pour le traitement de la livraison dynamique. Ainsi, le débit renvoyé par l'API CMS sera toujours supérieur à ce qui est spécifié dans le profil d'ingestion, à moins que le débit de la vidéo source ne soit inférieur à cela.
