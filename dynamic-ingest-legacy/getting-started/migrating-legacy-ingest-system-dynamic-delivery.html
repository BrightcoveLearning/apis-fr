introduction

Si vous souhaitez comprendre le fonctionnement de Dynamic Delivery et les avantages de son utilisation, consultez le Présentation de la livraison dynamique. Cette rubrique couvre certaines choses que vous devrez faire après le passage à Dynamic Delivery, en particulier en ce qui concerne l'utilisation de Dynamic Ingest.

Retranscoder des vidéos

Les vidéos existantes ne seront pas automatiquement converties ; contactez votre responsable de compte pour savoir comment transférer vos anciennes vidéos.

Si vous faites convertir vos vidéos existantes ou les retranscoder vous-même pour les déplacer vers la diffusion dynamique, notez que les métadonnées suivantes pour les vidéos changeront :

  • Les update_at Date
  • Chemins d'accès aux images miniatures et affiches (notez que nous recommandons ne pas coder en dur les URL des ressources vidéo dans vos pages et applications en général)

GDN

Si vous protégez vos vidéos avec DRM, l'un des avantages de Dynamic Delivery est que vous ne pas besoin de profils personnalisés pour appliquer DRM. Au lieu de cela, tout ce que vous avez à faire est de contacter le support et de configurer votre compte pour les types de DRM que vous souhaitez appliquer à vos vidéos. Après cela, toutes les vidéos ingérées ou retranscodées à l'aide d'un profil de diffusion dynamique seront automatiquement protégées. Voir Ingérer du contenu avec DRM pour plus de détails.

Partage médiatique

Sachez que la facturation fonctionne différemment pour le partage de médias dans Dynamic Delivery. Les coûts associés aux médias partagés sont facturés au compte principal auquel le média est partagé de, pas le compte sur lequel le média est partagé.

Notifications

Les notifications pour les ingestions de livraison dynamique sont similaires à celles pour l'acquisition héritée, mais il existe quelques différences. Si vous avez des applications ou des intégrations qui dépendent des notifications d'ingestion dynamique pour déclencher des actions ultérieures, vous devrez probablement les modifier un peu.

Vous trouverez ci-dessous des exemples de notifications pour l'achèvement réussi du rendu pour la livraison dynamique qu'une livraison non dynamique ingère pour mettre en évidence les différences.

Notifications de livraison dynamique et de livraison non dynamique
Succès du rendu de la livraison dynamique Succès du rendu de la livraison non dynamique 
    {
        "status": "SUCCESS",
        "entity": "5123312711001",
        "entityType": "DYNAMIC_RENDITION",
        "version": "1",
        "action": "CREATE",
        "jobId": "48c3a283-40f9-4214-8c27-21e3a040ca6a",
        "videoId": "5123312711001",
        "dynamicRenditionId": "default/audio64",
        "accountId": "57838016001"
    }
 
    {
        "entityType": "ASSET",
        "status": "SUCCESS",
        "version": "1",
        "jobId": "93d40db1-df3f-4023-bab6-51459cdda470",
        "videoId": "5122632511001",
        "profileRefId": "mp2",
        "entity": "5122644754001",
        "action": "CREATE",
        "accountId": "57838016001"
    }

Comme vous pouvez le voir, les différences ici sont que le entity_type pour l'ingestion de la diffusion dynamique est DYNAMIC_RENDITION à la place de ASSET , et vous obtenez un dynamicRenditionId au lieu d'une profileRefId.

Si vous avez une application ou une intégration qui dépend des notifications, vous devrez donc probablement apporter quelques petits ajustements à ce que vous recherchez dans les notifications.

Joueurs

Bonne nouvelle : si vous utilisez les codes d'intégration de lecteur standard ou les SDK de lecteur natifs, vous n'aurez pas besoin d'apporter de modifications aux lecteurs existants pour la livraison dynamique. Vos vidéos continueront de fonctionner comme avant.
